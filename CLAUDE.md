# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A single-page marketing site for VetCare (a veterinary clinic SaaS), in Bulgarian. Static HTML/CSS/JS ‚Äî no framework, no bundler-driven templating, no backend. All content lives in one `index.html`.

**All source lives in the `src/` directory ‚Äî `src/` is the project root.** `index.html`, `assets/`, `package.json`, `sitemap.xml`, `robots.txt`, etc. all live under `src/`, not the repo root. Run every `npm` command from `src/`, and read every path in this document as relative to `src/` unless stated otherwise (e.g. `assets/js/main.js` means `src/assets/js/main.js`).

## Commands

```bash
cd src              # source + package.json live here ‚Äî run all npm scripts from src/
npm install        # one-time, installs esbuild and sharp
npm run build       # minify assets/js/main.js -> main.min.js and assets/css/styles.css -> styles.min.css
npm run watch        # re-minify both JS and CSS on every save (via scripts/watch.js, esbuild's JS API in one process)
```

There is no test suite, lint config, or dev server in this repo. `index.html` references the `.min.` files directly, so **run `npm run build` (or keep `npm run watch` running) after editing `main.js` or `styles.css`**, or the live page won't reflect the change. To preview, just open `index.html` in a browser (or serve the directory statically).

`npm run watch` can't just be `esbuild --watch` on two entry points at once (the CLI's `--watch` only supports a single `--outfile`), so `scripts/watch.js` sets up two esbuild watch contexts side by side in one Node process instead of relying on shell job control (`&`), which isn't portable between the Windows and POSIX shells contributors may use.

`sharp` is a devDependency but has no npm script ‚Äî it's invoked ad hoc (`node -e "require('sharp')..."`) for one-off image work (recompressing PNGs, generating `.webp` companions). No image build pipeline exists.

## Architecture

- `index.html` ‚Äî the entire page, section by section (hero, problem, how-it-works, benefits, story, waitlist, pricing, demo, FAQ, contact, footer). Sections are HTML comments (`<!-- HERO -->`, etc.) ‚Äî use these as anchors when navigating.
- `assets/js/main.js` ‚Äî a single Alpine.js component (`Alpine.data('vetcare', ...)`) bound at `<body x-data="vetcare">`. All interactive state (mobile menu, scroll state, FAQ accordion, the three demo/waitlist/contact forms) lives in this one object. `submitDemo`/`submitWait`/`submitContact` validate, then POST to Web3Forms (see "Form submissions" below) and flip a `*Done`/`*Err`/`*Loading` flag based on the result.
- `assets/css/styles.css` ‚Äî one file, organized into clearly delimited banner-comment sections (`/* ‚îÄ‚îÄ Header ‚îÄ‚îÄ */`, `/* ‚îÄ‚îÄ Hero ‚îÄ‚îÄ */`, etc.) mirroring the HTML sections in the same order. Find the right section by matching the HTML comment, then look for the corresponding CSS banner.
- `assets/js/main.min.js`, `assets/css/styles.min.css` ‚Äî generated by esbuild. Never hand-edit; they're overwritten by `npm run build`.
- External dependencies are loaded via CDN in `<head>`: Alpine.js (`cdn.jsdelivr.net/npm/alpinejs`), Tabler Icons webfont, Google Fonts (Golos Text, Lora). No local copies, no package.json entries for these.
- SEO/sharing metadata (OpenGraph, Twitter cards, JSON-LD `@graph` with `SoftwareApplication`/`Organization`/`FAQPage`) is inline in `<head>`. The FAQ JSON-LD block and the visible FAQ section (`#faq`) list the same Q&A pairs ‚Äî keep them in sync if either changes. `og:image`/`twitter:image` and `sitemap.xml`'s `<image:loc>` have drifted from actual files before (referenced a `.webp` that was never committed) ‚Äî when touching these, verify the referenced file actually exists in `assets/images/`.

## Form submissions

- The demo, waitlist, and contact forms send to `hello@vetcare.bg` via [Web3Forms](https://web3forms.com), not a custom backend (there isn't one). All destination/config values ‚Äî target email, endpoint URL, access key ‚Äî live in one place: the `VETCARE_CONTACT` object at the top of `assets/js/main.js`. Change delivery settings there, not per-handler.
- Web3Forms was chosen over Zoho Forms (the business's actual mailbox provider) because Zoho Forms has documented CORS restrictions on `fetch()`-based AJAX submission from external origins ‚Äî there's no way to get a real success/failure signal back without a full-page redirect or a hidden-iframe workaround, which would break the existing inline `*Done` success-panel UX. Don't reintroduce Zoho Forms for this without re-solving that.
- `VETCARE_CONTACT.web3formsAccessKey` ships as the literal placeholder `'REPLACE_WITH_WEB3FORMS_ACCESS_KEY'` ‚Äî a real key (obtained at web3forms.com, verified against `hello@vetcare.bg`) has to be pasted in by a human with access to that inbox. Until then, every submission legitimately fails and shows the Bulgarian error message ‚Äî that's expected, not a bug to chase.

## Images

- No automated image pipeline. When adding/replacing a large raster photo (not an icon), manually: recompress with `sharp` (`png({ compressionLevel: 9, effort: 10, adaptiveFiltering: true })` ‚Äî lossless, no `palette: true` quantization, which barely shrinks photographic/alpha images further but risks banding), generate a `.webp` companion (`webp({ quality: 85 })` is a good default), and serve both via `<picture><source type="image/webp" srcset="...webp">` + `<img src="....png">` fallback (see the STORY section's `rex-kolaj` image for the pattern).
- Keep an `<img>`'s `width`/`height` attributes equal to the file's real intrinsic pixel dimensions (not a placeholder/aspirational size) ‚Äî CSS elsewhere uses `width:100%;height:auto`, so these attributes only set the reserved aspect ratio; a mismatch causes layout shift on load.
- On Windows, `convert` on PATH resolves to the OS's built-in disk-conversion utility, not ImageMagick ‚Äî don't use it for image work.

## Conventions

- **No inline styles.** All CSS goes in `styles.css` under a `vc-` prefixed class ‚Äî never add `style=""` attributes in HTML. The one accepted exception already in use is `style="display:none"` paired with `x-show`, which exists purely as a pre-hydration flash guard for Alpine (so the element is hidden before Alpine initializes); don't use inline `style` for anything else, including new instances of this pattern if a CSS-class-based alternative is reasonable.
- Class names are BEM-ish and consistently `vc-*` (e.g. `vc-hero-section`, `vc-mock-appt--dashed`). Match this prefix and the modifier-suffix (`--variant`) style for anything new.
- Alpine directives (`x-show`, `x-model`, `@click`, `:class`) are used throughout instead of vanilla JS DOM manipulation ‚Äî extend behavior through the `vetcare` Alpine component in `main.js`, not by adding separate `<script>` blocks or querySelector-based JS.
- Copy is in Bulgarian (`lang="bg"`); keep new user-facing text in Bulgarian and consistent in tone with existing copy.
