# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A single-page marketing site for VetCare (a veterinary clinic SaaS), in Bulgarian. Static HTML/CSS/JS — no framework, no bundler-driven templating, no backend. All content lives in one `index.html`.

## Commands

```bash
npm install        # one-time, installs esbuild and sharp
npm run build       # minify assets/js/main.js -> main.min.js and assets/css/styles.css -> styles.min.css
npm run watch        # re-minify both JS and CSS on every save (via scripts/watch.js, esbuild's JS API in one process)
```

There is no test suite, lint config, or dev server in this repo. `index.html` references the `.min.` files directly, so **run `npm run build` (or keep `npm run watch` running) after editing `main.js` or `styles.css`**, or the live page won't reflect the change. To preview, just open `index.html` in a browser (or serve the directory statically).

`npm run watch` can't just be `esbuild --watch` on two entry points at once (the CLI's `--watch` only supports a single `--outfile`), so `scripts/watch.js` sets up two esbuild watch contexts side by side in one Node process instead of relying on shell job control (`&`), which isn't portable between the Windows and POSIX shells contributors may use.

`sharp` is a devDependency but has no npm script — it's invoked ad hoc (`node -e "require('sharp')..."`) for one-off image work (recompressing PNGs, generating `.webp` companions). No image build pipeline exists.

## Architecture

- `index.html` — the entire page, section by section (hero, problem, how-it-works, benefits, story, waitlist, pricing, demo, FAQ, contact, footer). Sections are HTML comments (`<!-- HERO -->`, etc.) — use these as anchors when navigating.
- `assets/js/main.js` — a single Alpine.js component (`Alpine.data('vetcare', ...)`) bound at `<body x-data="vetcare">`. All interactive state (mobile menu, scroll state, FAQ accordion, the three demo/waitlist/contact forms) lives in this one object. There is no client-side submission wiring yet — `submitDemo`/`submitWait`/`submitContact` just validate and flip a `*Done` flag; there's no fetch/API call.
- `assets/css/styles.css` — one file, organized into clearly delimited banner-comment sections (`/* ── Header ── */`, `/* ── Hero ── */`, etc.) mirroring the HTML sections in the same order. Find the right section by matching the HTML comment, then look for the corresponding CSS banner.
- `assets/js/main.min.js`, `assets/css/styles.min.css` — generated by esbuild. Never hand-edit; they're overwritten by `npm run build`.
- External dependencies are loaded via CDN in `<head>`: Alpine.js (`cdn.jsdelivr.net/npm/alpinejs`), Tabler Icons webfont, Google Fonts (Golos Text, Lora). No local copies, no package.json entries for these.
- SEO/sharing metadata (OpenGraph, Twitter cards, JSON-LD `@graph` with `SoftwareApplication`/`Organization`/`FAQPage`) is inline in `<head>`. The FAQ JSON-LD block and the visible FAQ section (`#faq`) list the same Q&A pairs — keep them in sync if either changes. `og:image`/`twitter:image` and `sitemap.xml`'s `<image:loc>` have drifted from actual files before (referenced a `.webp` that was never committed) — when touching these, verify the referenced file actually exists in `assets/images/`.

## Images

- No automated image pipeline. When adding/replacing a large raster photo (not an icon), manually: recompress with `sharp` (`png({ compressionLevel: 9, effort: 10, adaptiveFiltering: true })` — lossless, no `palette: true` quantization, which barely shrinks photographic/alpha images further but risks banding), generate a `.webp` companion (`webp({ quality: 85 })` is a good default), and serve both via `<picture><source type="image/webp" srcset="...webp">` + `<img src="....png">` fallback (see the STORY section's `rex-kolaj` image for the pattern).
- Keep an `<img>`'s `width`/`height` attributes equal to the file's real intrinsic pixel dimensions (not a placeholder/aspirational size) — CSS elsewhere uses `width:100%;height:auto`, so these attributes only set the reserved aspect ratio; a mismatch causes layout shift on load.
- On Windows, `convert` on PATH resolves to the OS's built-in disk-conversion utility, not ImageMagick — don't use it for image work.

## Conventions

- **No inline styles.** All CSS goes in `styles.css` under a `vc-` prefixed class — never add `style=""` attributes in HTML. The one accepted exception already in use is `style="display:none"` paired with `x-show`, which exists purely as a pre-hydration flash guard for Alpine (so the element is hidden before Alpine initializes); don't use inline `style` for anything else, including new instances of this pattern if a CSS-class-based alternative is reasonable.
- Class names are BEM-ish and consistently `vc-*` (e.g. `vc-hero-section`, `vc-mock-appt--dashed`). Match this prefix and the modifier-suffix (`--variant`) style for anything new.
- Alpine directives (`x-show`, `x-model`, `@click`, `:class`) are used throughout instead of vanilla JS DOM manipulation — extend behavior through the `vetcare` Alpine component in `main.js`, not by adding separate `<script>` blocks or querySelector-based JS.
- Copy is in Bulgarian (`lang="bg"`); keep new user-facing text in Bulgarian and consistent in tone with existing copy.
